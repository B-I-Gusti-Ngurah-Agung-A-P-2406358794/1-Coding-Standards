## Module 3
### SOLID Principles Reflection

1) Pada modul ini saya menerapkan kelima prinsip SOLID pada dua bagian utama aplikasi, yaitu fitur produk dan fitur mobil (car). Untuk **Single Responsibility Principle (SRP)**, saya memastikan setiap kelas hanya punya satu tanggung jawab yang jelas: `ProductController` hanya mengatur alur HTTP dan view untuk produk, `ProductServiceImpl` hanya mengatur logika bisnis produk, sedangkan `ProductRepositoryImpl` hanya mengelola penyimpanan in-memory untuk produk; pola yang sama juga saya terapkan pada `CarController`, `CarServiceImpl`, dan `CarRepositoryImpl`. Untuk **Open/Closed Principle (OCP)**, saya memperkenalkan abstraksi repository di kedua domain, yaitu `ProductRepository` dan `CarRepository`, sehingga jika nanti ingin menambah cara penyimpanan baru (misalnya beralih dari list in-memory ke database), saya cukup membuat implementasi baru tanpa mengubah kode di service atau controller yang sudah ada. Untuk **Liskov Substitution Principle (LSP)**, saya menghilangkan pewarisan antara controller produk dan mobil sehingga tidak ada lagi relasi “is-a” yang tidak tepat; sekarang `ProductController` dan `CarController` berdiri sendiri dan bisa dipakai sesuai konteksnya tanpa mengubah perilaku yang diharapkan. Untuk **Interface Segregation Principle (ISP)**, saya menjaga agar interface seperti `ProductService`, `CarService`, `ProductRepository`, dan `CarRepository` tetap kecil dan fokus pada operasi CRUD yang benar-benar dibutuhkan oleh masing-masing fitur. Untuk **Dependency Inversion Principle (DIP)**, saya membuat modul level tinggi seperti `ProductServiceImpl`, `CarServiceImpl`, `ProductController`, dan `CarController` bergantung pada abstraksi (`ProductRepository`, `CarRepository`, `ProductService`, `CarService`) melalui constructor injection, bukan langsung pada detail implementasi, serta memindahkan detail penyimpanan ke kelas-kelas implementasi seperti `ProductRepositoryImpl` dan `CarRepositoryImpl`. Saya juga menyesuaikan model `Product` dan `Car` agar menyediakan akses properti yang jelas untuk digunakan oleh lapisan repository dan service.

2) Penerapan prinsip SOLID membuat proyek ini lebih mudah dikembangkan, diuji, dan dipelihara untuk kedua fitur, baik produk maupun mobil. Sebagai contoh, karena SRP dan DIP, saya bisa menulis unit test untuk `ProductServiceImpl` dan `CarServiceImpl` dengan mem‐mock `ProductRepository` dan `CarRepository` tanpa perlu tahu detail bagaimana data disimpan; ini menurunkan coupling dan membuat test seperti `ProductServiceImplTest` dan `ProductRepositoryTest` di test tetap sederhana meskipun repository dipecah menjadi interface dan implements. Dengan OCP dan abstraksi repository, jika suatu saat data produk dan mobil dipindah dari list in-memory di `ProductRepositoryImpl` dan `CarRepositoryImpl` ke database sungguhan, saya cukup menambah implementasi baru tanpa menyentuh `ProductController`, `CarController`, atau kelas service, sehingga risiko bug berkurang. LSP dan ISP membantu menjaga desain tetap konsisten dan mudah dipahami. Dengan itu, controller tidak lagi saling inherit secara paksa, setiap interface repository dan service hanya show method yang relevan, dan pemanggilan di controller menjadi lebih jelas karena selalu melalui interface yang terfokus.

3) Jika prinsip SOLID tidak diterapkan, proyek kecil seperti ini akan cepat menjadi sulit dikembangkan, terutama ketika fitur produk dan mobil mulai berkembang. Tanpa SRP dan DIP, `ProductController`, `CarController`, `ProductServiceImpl`, dan `CarServiceImpl` berisiko mencampur logika web, bisnis, dan penyimpanan data sehingga perubahan cara penyimpanan atau tampilan (misalnya menambah kolom baru di `Product` atau `Car`) mengharuskan pengubahan banyak kelas yang saling bergantung erat. Ini meningkatkan risiko bug dan menyulitkan penulisan test karena terlalu banyak dependensi yang harus disiapkan secara manual. Mengabaikan OCP pada repository akan memaksa kita untuk terus mengubah kelas seperti `ProductRepositoryImpl` atau `CarRepositoryImpl` setiap kali ada strategi penyimpanan baru, sehingga code lama mudah sekali rusak tanpa sengaja. Pelanggaran LSP dan ISP, misalnya dengan mempertahankan pewarisan tidak tepat antara controller produk dan mobil atau menaruh terlalu banyak tanggung jawab dalam satu interface service/repository, akan membuat kode membingungkan dan open to risk. Subclass atau implementasi dapat mengubah perilaku dengan cara yang tidak terduga, client dipaksa bergantung pada method yang tidak mereka perlukan, dan refactoring di masa depan (misalnya pemisahan fitur produk dan mobil ke modul terpisah) menjadi jauh lebih berisiko.

### Reflection 1
In this project, I implemented the edit and delete product features using Spring Boot and followed several clean code principles that I learned in this module. I tried to keep the code simple and easy to read by separating the controller, service, and repository layers so each class has a clear responsibility. I also used meaningful method names like update, and delete so the purpose of each function is easy to understand. In addition, I avoided repeating logic by placing business processes inside the service layer instead of directly in the controller. For secure coding, I used POST requests for actions that change data such as edit and delete, and I avoided exposing internal data unnecessarily. I also added basic checks, such as redirecting to the product list page when a product is not found, to prevent errors.

However, after reviewing the code again, I noticed some parts that can still be improved. The repository currently stores data in a simple list without validation or persistence, which is not safe for real applications. The code also does not include proper error handling or input validation, so invalid data could still be submitted. In the future, this could be improved by adding validation, better exception handling, and using a real database instead of in-memory storage. Overall, this exercise helped me understand how clean structure, clear naming, and safe request handling can make code easier to maintain and more secure.
